# Multilingual / Chinese Support

This document tracks Tingly-Wiki's support for non-English content, mixed-language content, and cross-language workflows (for example, English source material queried in Chinese).

## Current support matrix

| Scenario | Status | Notes |
|---|---|---|
| English ingest + English query | вЬЕ Fully supported | Original target. |
| Chinese ingest + Chinese query (BM25) | вЬЕ Supported | CJK bigram tokenizer (`MixedScriptTokenizer`). Recall is approximate but usable. |
| Mixed Chinese-English content (e.g. "OpenAIеЕђеПЄ") | вЬЕ Supported | Both scripts indexed independently. |
| English source + Chinese query | вЪ†пЄП Use vector search | BM25 cannot bridge languages. Configure a `VectorIndex`; LLM embeddings are natively multilingual. |
| Chinese source + English query | вЪ†пЄП Use vector search | Same as above. |
| Chinese page filenames | вЬЕ Supported | Unicode letters/digits preserved in `sanitizeName`. |
| LLM responds in user's question language | вЬЕ Supported | Prompt instructs the model to detect query language. |
| Multi-byte excerpts and truncation | вЬЕ Safe | Truncation happens on rune boundaries. |
| Cross-language entity merge ("OpenAI" вЖФ "OpenAIеЕђеПЄ") | вЭМ Not supported | Entities matched by exact string. See "Known limitations". |
| Chinese relationship extraction (regex link patterns) | вЭМ Not supported | `link_extractor.go` regexes only match `[A-Z][a-zA-Z]+`. See "Known limitations". |
| `Language` field on Source / Page | вЭМ Not supported | Schema has no language metadata. |

## What changed in the i18n update

| Concern | File | Change |
|---|---|---|
| Tokenizer dropped all CJK input | `index/tokenizer.go`, `index/fulltext.go` | New `Tokenizer` interface with default `MixedScriptTokenizer` (ASCII word splitting + CJK bigrams + Unicode letter handling). |
| Excerpt window sliced multi-byte chars | `index/fulltext.go` | `findExcerpt` snaps boundaries to UTF-8 rune starts. |
| Context assembler truncated by byte | `wiki/assembler.go` | `truncateStr` now counts runes. |
| `sanitizeName` flattened non-ASCII to `-` | `config/layout.go` | Preserves Unicode letters/digits across all scripts; only path-unsafe characters are replaced. |
| Prompts assumed English | `llm/prompts.go` | Query prompt responds in the user's language; extract/summarize/consolidate write in the source's language and preserve proper nouns. |

## Recommended usage

### Best-case setup

For mixed or non-English knowledge bases, configure both indexes:

```go
cfg := &config.Config{
    Storage:     storage.NewMemoryStorage(),
    LLM:         llm,
    VectorIndex: index.NewMemoryVectorIndex(...), // recommended for cross-language recall
    Layout:      config.DefaultLayout(),
}
```

`VectorIndex` provides cross-language recall via embeddings; `FullTextIndex` provides exact-match recall within a single script. Together they cover both within-language and cross-language queries.

### Custom tokenizer

If the default bigram is too noisy or if you want dictionary-based segmentation, implement `index.Tokenizer` and inject it:

```go
fti := index.NewFullTextIndexWithTokenizer(myJiebaTokenizer{})
```

Possible future implementations (not shipped):

- `JiebaTokenizer` вАФ wraps `gojieba` for dictionary-based Chinese segmentation
- `LLMTokenizer` вАФ calls the LLM to segment text (slow, accurate)
- `ICUTokenizer` вАФ ICU-backed multilingual segmentation (CGo)
- `EnglishOnlyTokenizer` вАФ restores the previous ASCII-only behavior

## Known limitations (scoped for follow-up work)

1. **No `Language` field on `Source` / `Page` / `Frontmatter`.** Language is inferred per-call by the LLM. A schema-level field would let the wiki short-circuit detection and constrain output explicitly.
2. **Entity dedup is exact-string.** "OpenAI", "openai", and "OpenAIеЕђеПЄ" are treated as three different entities. Adding alias matching (Unicode case-folding, NFC normalization, optional LLM-assisted merging) would fix cross-mention deduplication.
3. **`link_extractor.go` regex patterns are English-only.** Relationships such as "еЉ†дЄЙ еЬ® йШњйЗМеЈіеЈі еЈ•дљЬ" are silently ignored. Adding Chinese pattern templates is the next step here.
4. **`config.Config` has no tokenizer hook.** Custom tokenizers must be wired into `FullTextIndex` directly. A `Tokenizer` field on `Config` would make it easy to swap from the top.
5. **No Unicode normalization (NFC/NFD).** Pre-composed and decomposed forms of the same character are treated as different tokens.
